# 项目定位与复用方式

## 1. 仓库定位

本仓库不是某个单独业务系统，而是一个“后台管理标准底座”。

它负责提供：

- 可直接运行的后台 DEMO
- 一套稳定的页面骨架和视觉语言
- 一套统一的接口组织方式
- 一套可以复制到新项目里的规范文档

## 2. 适用场景

适合以下项目作为起点：

- 管理后台
- 运营后台
- 审批后台
- 配置中心
- 数据看板型中后台

不建议直接用于：

- C 端营销页面
- 强品牌定制官网
- 复杂编辑器类产品

## 3. 复用策略

### 3.1 直接复制项目

适用于：

- 新建一个结构相近的后台项目
- 希望保留当前布局、组件风格和开发方式

步骤：

1. 复制仓库并重命名。
2. 如果复制后的仓库用于承载真实业务，先执行 `bash ./scripts/init-business-project.sh .`，安装业务项目版 AI skills；如需重跑，必须显式使用 `--force`。
3. 替换 README、项目名称、菜单名称。
4. 保留 `layout`、`styles`、`router`、`api` 基础结构。
5. 用真实业务页面替换 `views/examples`。

### 3.2 只复用规范

适用于：

- 团队已有现成技术栈
- 只想借用设计规范和文档标准

做法：

- 复制 `docs/standards`
- 复制 `docs/templates`
- 参考 `web/src/styles` 中的变量和页面模式

### 3.3 只复用前端 DEMO

适用于：

- 需要快速演示后台布局和控件效果
- 需要和业务方先确认页面结构

做法：

- 保留 `web`
- Mock 数据按新业务换皮
- 文档页更新为当前项目自己的说明

### 3.4 派生为业务项目后的处理规则

如果复制后的仓库用于承载真实业务，默认目标应是“业务化替换”，而不是继续保留模板展示状态。

必改项：

- `README.md`
- `docs/README.md`
- `docs/tech/*`
- `docs/api/*`
- 菜单结构、路由标题、页面标题、按钮文案、空状态文案
- Mock 示例数据、接口说明中的资源命名和字段说明
- 与当前业务不符的模块名、示例名词和说明文字

可保留项：

- `docs/templates/*` 作为项目继续复用的模板资产
- `docs/standards/*` 中仍适用于当前团队的技术、设计、代码规范
- `web/src/layout`、`web/src/styles`、`web/src/router` 等稳定骨架

应归档、删除或显式说明保留原因的内容：

- 与当前业务无关的 demo 描述
- 仍把当前仓库描述为“通用后台 DEMO”或“模板工程”的业务文档
- 与当前业务无关的页面截图说明、示例接口说明、示例业务字段

## 4. 模板主干应长期保持稳定的部分

- 技术栈
- 目录分层
- 布局骨架
- 设计 token
- 页面模式
- 文档目录结构

## 5. 应按项目替换的部分

- 菜单结构
- 业务页面
- API 模块
- Mock 数据
- 业务名词
- 项目级 PRD 与交付文档
- AI skills 上下文与提示词模板
