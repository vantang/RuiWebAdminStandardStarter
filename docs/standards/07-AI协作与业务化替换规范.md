# AI 协作与业务化替换规范

## 1. 目标

本规范用于约束 AI 在本仓库及其派生业务项目中的工作方式，避免只完成页面换皮，却遗漏 README、技术文档、接口文档和模板残留清理。

## 2. 仓库身份判定

AI 开始工作前，必须先判断当前仓库属于哪一类：

- 模板仓库：用于维护通用后台底座、规范、示例页面和模板资产
- 业务项目：由模板复制而来，当前仓库承载真实业务逻辑和业务文档

如果当前仓库是业务项目，则默认目标是“业务化替换”，不是“保留模板展示状态”。

## 3. 开工前必做事项

在业务项目仓库中，AI 开工前至少要完成以下动作：

1. 阅读 `README.md`、`docs/README.md`、相关 `docs/standards/*`、相关 `docs/templates/*` 和本地 `.codex/skills/*`
2. 结合需求附件、原型、HTML/CSS、接口草案，判断受影响的页面、路由、API、Mock 和文档
3. 先输出“受影响文件清单”
4. 再输出“文档替换矩阵”

如果业务项目仓库尚未安装本地业务版 skills，建议先执行：

```bash
bash ./scripts/init-business-project.sh .
```

脚本默认只允许首次执行一次；如需强制重装，必须显式使用：

```bash
bash ./scripts/init-business-project.sh --force .
```

## 4. 业务项目必改范围

如果当前仓库已承载真实业务，以下内容默认进入 AI 的完成范围：

- `README.md`
- `docs/README.md`
- `docs/tech/*`
- `docs/api/*`
- `docs/qa/*`
- 页面标题、菜单名称、按钮文案、空状态文案
- 路由标题、业务字段、模块说明
- Mock 示例数据、接口命名、返回示例说明

以下内容可保留为团队复用资产，但必须明确其“模板/规范”身份：

- `docs/templates/*`
- 适用于当前团队的 `docs/standards/*`
- 稳定的布局、样式 token、页面模式与路由骨架

## 5. 模板残留定义

出现以下情况之一，应视为模板残留，必须清理或说明保留原因：

- README 或业务文档仍把当前仓库描述为“通用后台 DEMO”“模板工程”“示例项目”
- 页面、菜单、空状态、接口说明仍使用与当前业务无关的示例名词
- `docs/tech`、`docs/api` 与当前代码实现不一致
- 模板示例截图、字段、模块名称被误写成当前业务事实

## 6. 推荐执行流程

1. 用 `docs/templates/业务项目初始化清单.md` 明确仓库身份和必改项
2. 用 `docs/templates/AI业务项目启动提示词模板.md` 让 AI 先产出分析结果
3. 进入代码、Mock、文档同步实现
4. 用 `docs/templates/模板残留检查清单.md` 做收尾检查
5. 最终由 `$project-tester` 或等效验证流程补齐验证结果和未覆盖项

## 7. 最终汇报要求

AI 完成任务后，最终汇报至少应包含：

- 改动摘要
- 文档更新列表
- 模板残留检查结果
- 验证结果与未覆盖项

如果某些疑似受影响文件没有修改，必须在汇报中明确说明原因。
