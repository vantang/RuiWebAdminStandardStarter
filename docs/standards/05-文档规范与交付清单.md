# 文档规范与交付清单

## 1. 文档目录规范

- `docs/standards`：模板级规范
- `docs/tech`：实现方案和架构说明
- `docs/api`：接口文档
- `docs/qa`：测试与验收文档
- `docs/examples`：业务项目启动示例包
- `docs/templates`：模板文档、AI 提示词模板、初始化清单与检查清单

## 2. 一个完整业务后台项目最少应具备的文档

- 项目说明 README
- 页面/模块说明
- API 设计文档
- 技术方案文档
- 测试与验收文档
- 交付清单

## 3. 文档编写原则

- 先说明目标，再说明结构，最后说明细节
- 路径、模块名、接口名保持和代码一致
- 不用大段空话替代具体目录和约束
- 面向接手的人写，而不是面向作者自己写

## 4. 建议交付清单

- 可运行前端
- Mock 服务或联调方式
- 目录结构说明
- 接口说明
- 页面截图或页面清单
- 风险说明和后续建议

## 5. 业务项目文档替换矩阵

如果当前仓库已经派生为真实业务项目，建议至少覆盖以下文档替换：

- `README.md`：项目身份、模块范围、启动方式、验证方式、AI 使用入口
- `docs/README.md`：当前项目文档导航与推荐阅读顺序
- `docs/tech/*`：当前业务项目的实现方案、模块划分、集成边界
- `docs/api/*`：当前业务接口、字段约束、状态说明、联调说明
- `docs/qa/*`：测试环境、验收范围、回归结果和未覆盖项
- `docs/standards/*`：仍作为团队规范保留的部分，以及因当前项目而调整的约束
- `docs/examples/*`：仅作为示例参考，不应被误认为当前业务事实
- `docs/templates/*`：可继续保留为团队模板资产，但不应被误写成当前业务说明

建议优先复用以下模板资产完成首轮业务化文档：

- `docs/templates/业务项目README模板.md`
- `docs/templates/业务项目技术方案模板.md`
- `docs/templates/业务项目接口文档模板.md`
- `docs/templates/业务项目测试验收模板.md`
- `docs/templates/API设计模板.md`
- `docs/templates/测试验证模板.md`

如果需要查看“填好的参考答案”，可阅读：

- `docs/examples/README.md`

## 6. 推荐验收流程

业务项目建议按以下顺序执行测试与验收：

1. 准入检查：确认 `README`、`docs/tech`、`docs/api`、`docs/qa` 已形成首版文档
2. 冒烟检查：确认核心页面、关键接口、基础权限壳可运行
3. 核心业务流验证：覆盖列表、详情、表单、审批或导出等主路径
4. 边界与异常验证：覆盖空状态、失败反馈、权限不足、非法参数
5. 文档一致性检查：确认 `README`、`docs/tech`、`docs/api` 与实现一致
6. 模板残留检查：确认无与当前业务无关的 demo、示例或模板叙述
7. 回归与复测：对缺陷修复点和高风险区域做二次验证
8. 验收结论：明确“可验收 / 有条件验收 / 暂不建议验收”

## 7. 缺陷分级定义

业务项目的测试与验收文档，统一使用以下缺陷分级：

- `P0`：系统不可用、核心链路完全阻塞、数据错乱或存在严重安全风险，当前版本不可发布
- `P1`：核心业务流程无法完成或结果明显错误，虽非全站不可用，但当前版本不应验收通过
- `P2`：存在明显功能缺陷、交互异常或文档不一致，有明确绕过方式，可在有条件验收下记录后继续修复
- `P3`：低风险问题或优化项，不影响核心流程，可纳入后续版本处理

当验收文档引用 `P0 / P1 / P2 / P3` 时，默认按本定义执行，不再由项目成员各自解释。

## 8. AI 交付验收项

当 AI 参与业务项目改造时，交付结果至少应包含：

- 受影响文件清单
- 文档替换矩阵
- 文档更新列表
- 模板残留检查结果
- 验证结果与未覆盖项

同时应满足以下要求：

- `README`、`docs/tech`、`docs/api` 与代码实现保持一致
- `docs/qa` 记录当前测试范围、结论和未覆盖项
- 未修改但受影响的文件，需要说明不修改原因
- 不保留与当前业务不符的 demo、模板或示例叙述

## 9. 模板演进要求

- 每次模板主干发生明显变化时，至少更新一份规范文档
- 每次新增页面模式时，补充到设计规范
- 每次目录结构调整时，补充到技术规范
