# 技术栈与目录规范

## 1. 技术栈基线

当前模板统一使用：

- Vue 3
- Vite
- Element Plus
- Apache ECharts（用于数据可视化示例页）
- Vue Router
- Pinia
- Node.js 原生 HTTP Mock 服务

## 2. 技术选型原则

- 优先选择成熟、轻量、易维护的后台常用技术。
- 在没有明确收益之前，不为了“先进”而增加复杂度。
- 模板主干避免引入和具体业务强绑定的第三方库。

## 3. 目录规范

### 3.1 根目录

```text
.
├── docs
├── mock-server
├── scripts
└── web
```

### 3.2 前端目录

```text
web/src
├── api          # 接口封装
├── components   # 通用组件
├── layout       # 后台整体布局
├── router       # 路由定义
├── stores       # 全局状态
├── styles       # 变量与通用样式
├── utils        # 工具函数
└── views        # 页面视图
```

## 4. 命名规则

- 页面组件：`XxxView.vue`
- 通用组件：`PascalCase.vue`
- store：按领域命名，例如 `app.js`
- API 模块：按业务域拆分，不要把所有请求堆进一个文件
- CSS class：以语义为主，不以视觉结果命名

## 5. 路由组织原则

- 按页面能力分组，而不是按接口分组。
- 每个页面必须有稳定的 `meta.title`。
- 示例页面放在 `views/examples`，真实业务替换时可整体迁移。

## 6. Mock 服务组织原则

- Mock 只做演示和联调用途，不承载复杂业务逻辑。
- 路由尽量按资源组织，例如 `/api/customers`、`/api/projects`。
- 返回结构统一为：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```
