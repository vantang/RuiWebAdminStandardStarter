# 年终奖管理技术方案示例

## 1. 背景与目标

本项目用于支撑企业内部年终奖测算、审批和结果查询流程，面向 `HR 运营`、`部门负责人` 与 `财务复核人员` 提供统一的后台管理能力。

当前阶段目标：

- 完成年终奖方案配置、人员奖金测算与结果查询的基础闭环
- 支撑部门负责人在线确认和驳回异常奖金记录
- 为财务导出发放口径提供统一数据来源

本方案聚焦“年终奖管理模块首期建设”，不覆盖薪酬主数据维护、个税计算引擎和历史年度奖金归档。

## 2. 当前范围

本期纳入范围的模块：

- 奖励方案管理
- 奖金测算列表
- 奖金明细详情
- 审批与确认流程
- 导出记录查询

本期不纳入范围的内容：

- 跨年度奖金归档与对比分析
- 与外部薪酬系统的自动回写
- 个税与社保规则模拟

## 3. 总体结构

项目沿用模板仓库的基础结构：

```text
web           # 后台前端业务代码
mock-server   # 本地 Mock 与联调支撑
docs          # 项目文档、接口文档、交付物
site          # 如需对外展示再保留
```

协作关系：

```text
年终奖管理页面（web）
      |
      | HTTP / JSON
      v
Mock / 薪酬服务 / 审批服务
      |
      v
业务文档与接口文档（docs）
```

## 5. 模块与页面落点

| 模块 | 页面/能力 | 代码路径 | 路由 | 说明 |
| --- | --- | --- | --- | --- |
| 奖励方案管理 | 方案列表、方案表单 | `web/src/views/bonus/plan/` | `/bonus/plans` | 配置奖金规则、适用范围和发放口径 |

## 6. API 与 Mock 方案

接口组织原则：

- 页面不直接拼完整 URL
- 统一通过 `web/src/api/modules.js` 暴露接口
- 响应结构保持 `{ code, message, data }`
- 审批状态、奖金状态以接口返回为准，前端不自行推导最终状态

当前涉及的核心资源：

| 资源 | 接口路径 | Method | 说明 | Mock 状态 |
| --- | --- | --- | --- | --- |
| 奖励方案 | `/api/bonus/plans` | `GET / POST / PUT` | 方案列表与配置保存 | 已补 |
| 奖金测算 | `/api/bonus/calculations` | `GET` | 员工奖金测算列表 | 已补 |
| 奖金详情 | `/api/bonus/calculations/:id` | `GET` | 单条奖金明细详情 | 已补 |
| 审批动作 | `/api/bonus/approvals/:id/action` | `POST` | 确认或驳回奖金记录 | 已补 |
| 导出记录 | `/api/bonus/exports` | `GET / POST` | 导出任务列表与创建 | 已补 |

## 7. 关键交互与状态

重点业务规则：

1. 奖金测算结果生成后，只有 `待确认` 状态的数据允许部门负责人审批。
2. 被驳回的记录必须填写驳回原因，并进入 `待修正` 状态。
3. 已确认记录允许导出，但不允许再次编辑测算口径。

关键状态建议统一整理：

| 状态/字段 | 含义 | 来源 | 页面表现 |
| --- | --- | --- | --- |
| `bonusStatus` | 奖金状态 | 接口返回 | `待确认 / 已确认 / 待修正 / 已导出` Tag |
| `approvalResult` | 审批结果 | 接口返回 | 操作按钮显隐、详情轨迹 |
| `exceptionFlag` | 异常标记 | 接口返回 | 列表警示色、详情提示块 |

## 8. 联调与环境策略

- 本地开发：优先走 `mock-server`
- 联调阶段：切换到测试环境薪酬服务与审批服务
- 配置入口：通过 Vite 代理转发 `/api`
- 数据口径：后端返回最终奖金状态、审批状态与异常原因，前端只负责展示和交互约束

## 10. 风险、依赖与待确认项

风险：

- 奖金测算口径在 HR 与财务之间存在口径分歧，可能导致字段和状态调整
- 审批流若接入现有 OA 平台，接口时序和状态回写可能变化

依赖：

- HR 提供奖金规则与字段口径
- 财务确认导出口径
- 审批服务提供状态回写能力

待确认项：

- 奖金详情是否需要展示历史调档记录
- 导出任务是否支持按部门和批次分片导出
