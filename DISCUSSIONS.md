# GitHub Discussions Guide

## 目的

`GitHub Discussions` 适合承载“开放讨论”和“社区交流”，而不是把所有问题都塞进 Issue。

对本仓库而言，Discussions 的目标是：

- 承接开放式问答，而不是可执行缺陷单
- 承接方向性想法，而不是已经明确要做的需求卡片
- 承接经验分享、复用案例、模板展示，而不是代码合并流程
- 降低维护者在 Issue 区被大量“使用咨询”淹没的风险

## 什么时候用 Discussions

以下场景更适合发到 Discussions：

- “这个模板适不适合某类后台项目？”
- “这套目录结构我应该怎么裁剪？”
- “这个设计选择背后的原因是什么？”
- “我基于这个模板做了一个变体，想分享经验或截图”
- “我有一个方向性想法，想先讨论是否值得进入主干”
- “文档里某些部分不够清楚，想先讨论最佳写法”

## 什么时候不要用 Discussions

以下场景不要发到 Discussions：

- 可稳定复现的缺陷
- 已经足够明确、可以直接排期的功能需求
- 安全漏洞、敏感泄露、凭据暴露
- 已经完成实现、等待合并的代码改动

对应渠道如下：

- 缺陷：`Bug report` Issue
- 功能建议：`Feature request` Issue
- 使用帮助：优先 `Support request` Issue；如果你启用了 Discussions，也可以在 `Q&A` 分类先讨论
- 代码提交：Pull Request
- 安全问题：按 `SECURITY.md` 私下披露

## 建议的 Discussions 分类

如果你准备把这个仓库公开到 GitHub，并启用 Discussions，建议至少创建以下分类：

- `Q&A`
  用于使用问题、接入疑问、实现方式讨论
- `Ideas`
  用于方向性建议、模板演进想法、功能脑暴
- `Show and tell`
  用于展示基于本模板的项目、截图、实践经验
- `General`
  用于不属于以上类别的社区讨论
- `Announcements`
  仅维护者使用，用于发布重大变更、迁移通知、版本策略

## 建议的使用边界

为了避免 Discussions 和 Issues 混用，建议采用以下边界：

- `Discussions` 负责讨论、澄清、收集反馈
- `Issues` 负责可执行、可追踪、可关闭的工作项
- `PR` 负责具体改动和合并审查

一个常见流程是：

1. 先在 `Discussions` 里讨论方向或用法。
2. 当问题收敛到一个明确动作后，再转成 `Issue`。
3. 实现完成后，通过 `PR` 提交改动。

## 对提问者的建议

在 Discussions 里发帖时，建议尽量说明：

- 你的目标是什么
- 你目前处于什么阶段
- 你已经尝试过哪些方法
- 你参考过哪些文档、文件或页面
- 你希望得到的是方案讨论、路径建议，还是经验分享

如果你使用了 AI 辅助，请补充说明：

- 哪一部分由 AI 辅助完成
- 你已做过哪些人工复核
- 你想讨论的是 AI 建议本身，还是你整合后的实现路径

## 对维护者的建议

维护者在开启 Discussions 后，建议遵循以下原则：

- 对开放问题优先引导到合适分类
- 对已经明确成缺陷或功能项的讨论，及时转为 Issue
- 对安全相关内容立刻终止公开讨论，并引导到 `SECURITY.md`
- 对纯支持类、重复性问题，尽量沉淀成文档或 FAQ
- 对有代表性的讨论结论，补进 `README.md`、`SUPPORT.md` 或 `docs/standards`

## 推荐的社区分流规则

可以直接对外采用以下约定：

- 想讨论想法、问用法、分享案例：发 `Discussions`
- 想报告缺陷、提明确需求、请求可执行修改：发 `Issues`
- 想提交代码或文档改动：发 `PR`
- 想报告安全问题：不要公开发帖，改走私下安全披露

## 启用前检查

在 GitHub 仓库真正开启 Discussions 前，建议先确认：

- `README.md` 中已放入 Discussions 入口说明
- `SUPPORT.md` 已写清楚与 Issue 的分流规则
- 维护者知道如何把讨论结果转成 Issue 或 PR
- 不会把安全披露误导到公开讨论区

如果你把这个仓库作为模板再次分发，建议连同本文件一起保留，并按你的团队方式调整分类名称和协作边界。
