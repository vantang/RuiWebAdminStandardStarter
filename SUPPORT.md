# Support

## 支持范围

本仓库欢迎公开使用、Fork、二次开发，以及人类或 AI/Agent 参与的协作式开发。

维护者通常愿意处理或回答以下类型的问题：

- 如何启动、调试、构建和校验本项目
- 如何把本仓库作为后台模板进行复用
- 如何理解目录结构、示例页面、Mock 服务和文档规范
- 文档描述不清、示例不足、模板缺少必要说明
- 与开源协作流程相关的问题，例如提 Issue、提 PR、贡献规范

## 先自助排查

在请求支持前，建议先完成以下检查：

- 阅读 `README.md`
- 阅读 `CONTRIBUTING.md`
- 阅读 `SECURITY.md`
- 查看 `docs/standards`、`docs/tech` 和相关示例文档
- 本地执行 `npm run check:strict`
- 确认问题是否只存在于你自己的私有改造、私有环境或私有数据中

## 选择正确渠道

不同类型的问题请走不同入口：

- 可复现缺陷：使用 GitHub 的 `Bug report` Issue 模板
- 新能力建议：使用 GitHub 的 `Feature request` Issue 模板
- 使用帮助、接入疑问、复用咨询、文档缺口：使用 GitHub 的 `Support request` Issue 模板
- 如果你启用了 GitHub Discussions，开放式问答、经验分享和方向性想法也可以先进入 `Discussions`
- 安全漏洞、敏感泄露、凭据暴露：不要公开提交，请按 `SECURITY.md` 私下披露

## 提问时请尽量提供

为了减少来回追问，请尽量一次性给出：

- 你的目标是什么
- 你当前做了哪些改动
- 你参考了哪些文档或代码路径
- 你实际执行了什么命令
- 你期望发生什么
- 实际发生了什么
- 你的运行环境，例如操作系统、Node 版本、浏览器、部署方式
- 关键日志、报错、截图、最小复现步骤

如果你使用了 AI 辅助，请一并说明：

- 哪些内容由 AI 生成或修改
- 你已进行过哪些人工复核
- 问题出现在 AI 建议阶段还是人工整合阶段

## 维护者通常不会提供的支持

以下场景通常不在本仓库的常规支持范围内：

- 你的私有业务系统中的一对一实施服务
- 与本仓库无关的通用前端、后端或运维培训
- 明显脱离模板主干的大规模定制项目排障
- 无复现信息、无上下文、仅贴一句“不能用”的问题
- 涉及敏感数据、密钥、商业机密但又无法私下沟通的问题

## 响应预期

本仓库的支持通常是社区协作和维护者尽力而为，不构成服务等级承诺。

维护者可能会：

- 直接回答
- 要求补充复现信息
- 引导你改用更合适的 Issue 类型
- 建议你提交 PR 直接改进文档或模板
- 对超出范围的问题不作进一步处理

## 最佳实践

为了让支持更有效，建议：

- 一个 Issue 只讨论一个明确问题
- 标题中写清模块或主题
- 不要在公开 Issue 中粘贴密钥、令牌、内部地址或敏感数据
- 不要把安全问题当普通 Bug 公开提交
- 如果问题最终定位为文档缺失，欢迎直接补文档并提交 PR
